# Changelog

## [1.0.0-rc.4] - unreleased

### Added
- **`HeapsUtils.solidTile` / `solidBitmap` helpers** — single shared path for solid-color `h2d.Tile` / `h2d.Bitmap` creation. `solidTile(color, w, h)` returns a 1×1 GPU texture stretched to `(w, h)`; `solidBitmap(color, w, h, ?parent)` is the Bitmap convenience. Color is strict-D `0xAARRGGBB` — top byte is alpha (e.g. `0x80FF8000` = 50%-alpha orange). Top-byte=0 is treated as opaque to match Heaps' `Tile.fromColor` default and keep bare `0xRRGGBB` callers working; for fully transparent, call `Tile.fromColor` directly. Migrated call sites: `MAObject` debug hitbox overlays, `MultiAnimBuilder.generatePlaceholderBitmap` (`SolidColor` generated tile), text tile background, the incremental `SolidColor` update path, `ScreenManager` modal overlay, and the `ProgrammableCodeGen` `SolidColor`/`Cross` emitters (the legacy inline `if (c >>> 24 == 0) c |= 0xFF000000` bake — a no-op once Heaps masks the top byte — is deleted).
- **DevBridge `debugger()` API and `get_debugger_hits` RPC** — JS-`debugger;`-style breakpoint for game code. `screenManager.devBridge.debugger(data, pause=true)` snapshots `data`, auto-captures file/line/method via `haxe.PosInfos`, pushes an SSE `debugger` event, appends the hit to a ring buffer (capacity 100), and optionally pauses the game loop (shared `setPaused()` helper extracted from `handlePause`). Poll via new `get_debugger_hits` RPC (params: `clear`, `limit` ≤ 100, `since_id` cursor; returns `hits[]`, `total`, `dropped`, `lastId`). Compiled out in release — only active with `-D MULTIANIM_DEV`. 9 regression tests in `DevBridgeTest` covering empty buffer, capture without pause, id increment, `since_id` cursor (including past-latest), `limit`, `clear`, ring-buffer drop behavior, and paused-flag side-effect.
- **`UIInteractiveSource` interface** (`bh.ui.UIInteractiveSource`) — abstraction over anything that can feed interactive elements to `UIScreen.addInteractives` / `UIRichInteractiveHelper.register`. Four methods: `getInteractives():Array<MAObject>`, `addRebuildListener(fn)`, `removeRebuildListener(fn)`, `setParameter(name, value)`. Implemented by both `BuilderResult` (runtime path) and every `@:manim` codegen-generated instance class, so both paths are interchangeable at call sites. `UIScreen.addInteractives` and `UIRichInteractiveHelper.register`/`resync`/`bind` now accept `UIInteractiveSource` instead of `BuilderResult` — existing `BuilderResult` calls still work via polymorphism.
- **Codegen instances expose their interactives** — `@:manim` instance classes now generate `getInteractives():Array<MAObject>` (walks the instance's scene graph via `ProgrammableBuilder.collectInteractives`), `addRebuildListener`/`removeRebuildListener` (append/remove from internal `_rebuildListeners` array), a private `_fireRebuildListeners()` helper fired at the end of every typed setter (after `_applyVisibility` / `_updateExpressions`), and a generic `setParameter(name:String, value:Dynamic)` dispatcher that routes to the right typed setter (`setStatus` etc.). The enum dispatcher accepts either an Int index or a String enum value (e.g. `instance.setParameter("status", "hover")` looks up `"hover"` in the param's enum values at codegen time). Instance classes now declare `implements bh.ui.UIInteractiveSource` so `screen.addInteractives(codegenInstance, prefix)` works symmetrically to the runtime path. Switch arm flips inside codegen programmables auto-resync the screen's wrapper map through the same rebuild listener mechanism used by the runtime path. 7 integration tests in `UIRichInteractiveHelperTest` covering `getInteractives`, direct screen wiring, string-dispatched `setParameter`, full `UIRichInteractiveHelper` end-to-end cycle, autoStatus end-to-end, and `@switch` arm resync on `setMode()`.
- **`BuilderResult.addRebuildListener(fn)` / `removeRebuildListener(fn)`** — register a callback that fires once per `applyUpdates()` cycle in which any parameter changed. Use to resync external state (e.g. screen-side `UIInteractiveWrapper` maps) after a `@switch` arm flip or param-dependent `repeatable` count change. The listener fires AFTER all conditional/tracked updates and dynamic ref propagation, so the BuilderResult collections are coherent when it runs. Re-entrant `setParameter()` from inside a listener is supported (enters a fresh applyUpdates cycle). No-op on non-incremental results (static snapshots never rebuild), so callers can install listeners without branching on incremental mode.
- **`UIRichInteractiveHelper.resync(result, ?prefix, ?metadataKey)`** — resync bindings against the BuilderResult's current `interactives` list. Removes bindings whose ids no longer exist (filtered by prefix), adds bindings for new ids, and *preserves* existing bindings' state (does not reset `currentState` to Normal). Use after a SWITCH arm flip / REPEAT rebuild changes which interactives are present. The screen's auto-status helper is now resynced automatically (see "Fixed" below).
- **`@switch` block** — `@switch(param) { value: ...; default: ... }` groups multiple conditions on a single parameter. Supports single-value arms (`idle: ...`), multi-value with pipe (`hover | pressed: ...`), comparisons (`<= 10:`, `> 90:`), ranges (`0..100:`), `default:` fallback, and block arms (`value { ... }`) for multiple elements per case. Block arms can contain nested `@()` conditionals. Implemented as `SWITCH` NodeType with O(1) Haxe `switch` dispatch in codegen for pure-enum arms. No unnecessary scene graph wrappers — single-child arms toggle visibility directly on the child.
- **Conditional blocks** — `@(cond) { ... }`, `@else { ... }`, `@else(cond) { ... }`, `@default { ... }` — group multiple elements under a single condition without repeating the `@` prefix on each element. Wraps children in a POINT node. Supports nesting (block-in-block).
- **`@any`/`@all` conditional keywords** — `@any(p1=>v1, p2=>v2)` matches when ANY listed condition matches (OR logic). `@all(p1=>v1, p2=>v2)` matches when ALL listed conditions match (AND logic, same as `@()`/`@if()`). Unlisted parameters are ignored in both modes. Stacked conditionals (`@(a=>1) @(b=>2)`) now produce a clear error directing users to `@all()` or `@any()`.

- **Tile source `center()`/`pivot()` modifiers** — wrap any tile source to set the tile's origin point. `center(source)` is shorthand for `pivot(0.5, 0.5, source)`. `pivot(x, y, source)` sets an arbitrary pivot using 0–1 ratio coordinates. Works everywhere tile sources are used: `bitmap()`, particle `tiles:`, etc. When used inside `bitmap()`, overrides the bitmap's hAlign/vAlign. Implemented as `TSPivot` enum variant in `TileSource`, handled in builder, codegen, and all three bitmap creation paths.

- **Dynamic programmable references in `dynamicRef`** — `dynamicRef($template, val=>$hp)` where `$template` is a `string` or enum parameter of the enclosing programmable. The resolved parameter value names the target programmable. When the template parameter changes, the old visual is torn down and the new programmable is fully rebuilt. Forwarded parameters (`val=>$hp`) propagate incrementally without rebuild. Backward compatible: `$progName` where `progName` is not a parameter falls back to literal name. Strict validation: forwarded parameters must exist on the target programmable (RTE if missing, allowed if target has a default). Works in both builder (via `IncrementalUpdateContext.trackDynamicName`) and codegen (via `generateDynamicNameRefCreate` with container/name/result fields).

- **`UIEventPriority` constants and overlay priority** — `UIEventPriority` class with `Content` (0), `Overlay` (100), `Modal` (200) tiers for event priority. Dropdown (`UIStandardMultiAnimDropdown`) now implements `UIElementPriority` — sets `Overlay` when panel opens, `Content` when closed. `UIPanelHelper` sets `Overlay` priority on all panel interactives. `UIInteractiveWrapper.eventPriority` is now publicly writable for programmatic override. Fixes the dropdown click-through bug where background interactives received events through open overlays.
- **`ScreenShakeHelper`** — lightweight additive screen shake helper for impact feedback. Supports basic shake with linear decay, directional shake (horizontal recoil, vertical landing), and custom decay curves. Multiple concurrent shakes stack naturally. Non-destructive: applies offsets as per-frame deltas relative to the previous frame, so gameplay can freely move the target (camera scroll, layout, animation) without the shake fighting back to a captured baseline. Uniform jitter via `hxd.Rand`.
- **Particle `emitFilter`** — optional callback on `ParticleGroup` to filter particles by spawn position. Called after world position is computed; return `false` to discard the particle. Works for both relative and non-relative groups.
- **Particle `externallyDriven`** — when `externallyDriven: true`, particles do not auto-update from the render loop. Game code drives the simulation via `group.advanceTime(dt)` or `particles.advanceTime(dt)`. Follows the same pattern as `AnimationSM.externallyDriven`. Both group-level time (shutdown, attached paths, spawn curves) and per-particle physics are suppressed until manually advanced.
- **Grid `sceneToHex()`** — convert scene coordinates to hex grid coordinates. Returns the nearest hex `CellCoord` regardless of whether a cell exists. Works for hex grids only (returns null for rect).
- **`.anim` `flipX` / `flipY`** — flip all frames of an animation horizontally/vertically *in place*. The sprite keeps the same untrimmed screen footprint regardless of where the pivot sits within it (tiles are mirrored around the untrimmed center, trim offsets are recomputed, extrapoints auto-mirror around the center). Usable at the file level (`flipX: yes` before any animation), inside `animation { ... }` blocks, and in the `anim name(flipX:yes): "sheet"` shorthand. Animation-level setting overrides file-level default. Shared atlas tiles are cloned before flipping to avoid cross-instance mutation.
- **Particle `shutdown` block and API** — graceful particle stop for looping emitters. `shutdown: { duration: 1.0, curve: easeOutQuad, alphaCurve: ..., sizeCurve: ..., speedCurve: ... }` configures wind-down behavior in `.manim`. At runtime, `particles.shutdown()` or `group.shutdown(?duration, ?curve)` starts a timed shutdown that curves particle count (selectively stops recycling dead particles) and optionally applies global alpha/size/speed multipliers. All curves use progress convention (`mult = 1.0 - curve(t)`). Query state via `isShuttingDown()` and `getShutdownRate()`. `emitBurstAt()` still works during shutdown. Existing `onEnd()` fires when the last particle dies.

- **`@switch` incremental mode** — `setParameter()` on a `@switch`-controlled parameter now triggers full arm rebuild (teardown + rebuild). Arm content is built non-incrementally; all param refs across all arms are collected so that changing any referenced param (switch param, inner conditionals, repeatable counts, `$param` expressions, positions) triggers the rebuild. Supports nested `@switch`, `@()` conditionals, and repeatables inside arms.
- **`@switch` codegen lazy rebuild** — codegen no longer pre-builds all arms and toggles visibility. Instead, uses ordinal-based arm lookup (`switchCounter` at compile time, DFS walk at runtime) to rebuild only the active arm via `ProgrammableBuilder.rebuildSwitchArm()`. Always rebuilds on any parameter change, forwarding current param values to the builder for correct expression evaluation.
- **`@switch` duplicate default rejection** — parser now rejects `@switch` blocks with multiple `default` arms at parse time.

### Changed
- **BREAKING: Strict-D color semantics — `0x` literals are Heaps AARRGGBB, preserved verbatim** — `addAlphaIfNotPresent` is deleted. `ColorArg` abstract is deleted; `PPTColor` storage is a plain `Int`. The parser bakes alpha only for CSS `#` forms (3/6-digit shorthand); `0x...` literals pass through unchanged regardless of length, and every Int-consumer path (`setColor`, `setParameter`, `createFrom`, hot-reload, node tint, graphics shapes, atlas2 autotile, `PixelOutline.inlineColor`, `ReplacePaletteShader`) trusts its input. This fixes the BUG-3 footgun where `setColor(0)` was silently clobbered to `0xFF000000` (opaque black) and `#00000000` / `0x00000000` transparent was unreachable from runtime callers — transparent and opaque-black are now distinguishable everywhere. **Migration:** any `.manim` source using `0xRRGGBB` (6-hex Heaps literal) to mean "opaque RRGGBB" must change to `#RRGGBB` or `0xFFRRGGBB`. Automated sed: `sed -i -E 's/0x([0-9A-Fa-f]{6})\b/#\1/g' *.manim`. The 154 occurrences in this repo's test `.manim` files are migrated; cross-repo consumers (`hx-multianim-playground`, `hx-easy-game`) still have unmigrated literals and will render them transparent until their own sed passes run. Any Haxe caller passing `0xFF0000` to a color-accepting API expecting opaque red must change to `0xFFFF0000`. Existing Fixed entries on `setParameter` / `0xRRGGBB` alpha baking (below) documented the old compensation that is now replaced by this simpler "Int is final" rule. The `testCodegenColorParamTransparentFootgun` test is now `testCodegenColorParamTransparentPreserved` and asserts the fixed behavior.
- **`@ifstrict` removed** — replaced by `@all()` (AND on listed params, unlisted ignored). The old `@ifstrict` behavior (requiring ALL programmable params to be mentioned) is no longer available.
- **`layout(name)` single-argument form** — `repeatable($i, layout("name"))` is now the only accepted form. The legacy two-argument `layout("group", "name")` (where the first argument was a label-only group identifier) is rejected at parse time with a clear error directing users to drop the first argument. All bundled examples, tests, cookbook, and reference docs migrated.
- **Parser cleanup — hxparse removal** — `MultiAnimParser` is now a thin facade that delegates to `MacroManimParser`. The old hxparse-based lexer/parser, the `MPToken`/`MPKeywords`/`LayoutsParsingState`/`MPInterpolationEnum`/`IdentifierType`/`NumberType`/`ValueType` enums and associated machinery are deleted. A single hand-written recursive-descent parser now covers both compile-time (macro codegen) and runtime paths. Public API (`MultiAnimParser.parseFile()`, `ParametersDefinitions`, `PlaceholderTypes`, etc.) is unchanged.
- **`Particles.addGroup(g)` signature** — removed the unused optional `index` parameter (it was never honoured inside the method). Callers passing a second argument must drop it.
- **Hyperlink callback error re-throw in DEV mode** — rich text hyperlink callback errors are now re-thrown after tracing when `MULTIANIM_DEV` is defined, making callback bugs visible during development.
- **`@switch` parameter type support** — `@switch` single-value arms now route through the same type-aware converter as `@(param => value)`. `@switch` on `color` params accepts `#RRGGBB` literals, on `string` params accepts `"quoted string"` literals, on `bool` params accepts `true`/`false`, and on `int` params accepts bare/negative integers. Previously these either failed to tokenize (strings/colors) or silently produced `CoEnums` string-matches that never matched integer-backed values (bool, color) at runtime. Range/comparison arms (`<= N`, `N..M`, `> N`, etc.) are now rejected at parse time on non-numeric parameters with a clear error. `@switch` on `float`, `tile`, and `flags` parameters is rejected at parse time (use `@(p => bit[N])` for flag tests). Pipe-separated multi-value arms (`v1 | v2 | v3`) are type-aware for discrete non-enum/non-string params (color, int, uint, bool) via a new `CoAnyOf` conditional — each arm value is converted through `stringToConditional` and OR'd at match time. Enum/string pipe arms keep the existing `CoEnums` path.
- **`crossfade` transition semantics** — `crossfade(duration, ?easing)` is now a sequential blend-through-zero instead of a simultaneous two-sided fade. The outgoing element fades fully to alpha 0 over `duration`, then the incoming element waits and fades from 0 to its target alpha over another `duration`. Total wall-clock duration is `2 × duration`. Previously both states animated in parallel over a single `duration`, producing a noticeable mid-point alpha spike when old + new were both ~50% visible. Applies to all three transition paths (incremental builder, codegen no-parent, codegen parent-rewrite). Reference image for `95-transition` regenerated.

### Fixed
- **`generated(color(w, h, #c))` silently dropped explicit #RRGGBBAA alpha** — both the runtime (`MultiAnimBuilder.generatePlaceholderBitmap`) and codegen (`ProgrammableCodeGen` `SolidColor`/`Cross`) paths called `h2d.Tile.fromColor(color, w, h)` with no alpha arg. Heaps defaults alpha to 1.0 and masks the top byte of `color`, so parser-baked strict-D values like `#7fdbda33` (→ `0x337fdbda`, alpha `0x33`) rendered fully opaque — the author's explicit alpha byte was ignored. Same bug hit `.manim` `@final RGBA8 = #FF880080` constants used as `generated(color(...))` args, including those resolved through `$param` refs. Now both paths go through the shared `HeapsUtils.solidTile` helper which unpacks the alpha byte: `top = color >>> 24; alpha = top==0 ? 1.0 : top/255.0; Tile.fromColor(color & 0xFFFFFF, w, h, alpha)`. Top-byte=0 preserves the opaque default so legacy Haxe-side callers passing bare `0xRRGGBB` (e.g. `TileHelper.generatedRectColor(16, 16, 0x00FFFF)` in test code, `ScreenManager.ModalOverlayConfig.color`) still render opaque as before. Reference images for `#78 characterSheetDemo` (dividers written as `#7fdbda33` — 20% alpha cyan), `#107 rubeGoldberg` (`RGBA8 = #FF880080` / `graphics rect` with `#FF880080` — 50% alpha orange), and `#67 interactiveMetadata` (`MAObject` debug overlay alpha 0.5 re-encoded as byte 0x80, 1-LSB drift in quantization) regenerated.
- **Access violation when a programmable has two `@switch` blocks (or any second `@switch` following enough preceding siblings)** — `MacroManimParser.createNode` computed `uniqueNodeName` via `Std.string(type)`. For a `SWITCH(param, arms)` node the payload is a full subtree whose child `Node`s carry `parent` back-pointers up through the root programmable (and all its already-accumulated siblings). `Std.string` recurses the enum payload unbounded → HashLink stack overflow → `Access violation`. Only the second `@switch` tripped it because the first one's siblings hadn't yet grown the tree large enough. Fix: for `SWITCH` nodes only, `uniqueNodeName` now uses `Type.enumConstructor(type)` (short `"SWITCH"` identifier); all other node types keep the detailed `Std.string(type)` form to preserve uniqueness for `ProgrammableBuilder.buildNodeByUniqueName` / `findNodeByUniqueName` (REPEAT-child runtime forwarding). Regression coverage: new `test107_K_twoSwitchesWithBlockArmBug` and `test107_K_twoSwitchesOnDifferentParams` in `RubeGoldbergIncrementalTest`, plus the two restored `@switch` blocks in `rubeGoldberg.manim` exercise it in the visual test path.
- **`@final` constants at programmable-body scope lost after `setParameter`-triggered rebuild** — `IncrementalUpdateContext` deep-copies `indexedParams` at construction, which happens *before* `startBuild` evaluates `FINAL_VAR` nodes. Any `@final NAME = expr` declared at the programmable body was therefore missing from the context's persistent copy; the first `setParameter` call that triggered re-evaluation of an element referencing `$NAME` threw `reference NAME does not exist`. Fix: new `IncrementalUpdateContext.syncFinalsFromBuilder(live)` copies surviving `ExpressionAlias` entries from the live builder state into the context after initial build. Nested `@final`s (inside `point {}`, `repeatable`, etc.) are cleaned up by `cleanupFinalVars` before the sync runs and are correctly excluded — matching their lexical scope. Regression coverage: `test107_J_finalConstantsSurviveRebuild` plus the `@final SW / SH / RGB3 / RGB6 / RGBA8 / HEX_FULL / HEX_SHORT / TRANSPARENT` block in `rubeGoldberg.manim` referenced across multiple setter round-trips.
- **Codegen `setParameter` dispatcher Int-family coercion** — Int-family params (`int`, `uint`, `color`, `hexDir`, `gridDir`, `range`, `flags`) previously used an unchecked `cast _value` in the generated `setParameter(name, value)` dispatcher. On HL, passing a Dynamic `Float` (common when values come from JSON, arithmetic, or cross-target code) crashed on the downstream Int field write; on JS it silently passed the Float through, producing cross-target drift. Now coerces Dynamic input through the same pattern already used for `PPTFloat`: accepts Int directly, truncates Float via `Std.int`, parses String via `Std.parseInt` (throwing on unparseable), rejects other types. `PPTString` similarly gained an `Std.isOfType(_value, String)` guard to reject Int/Float/Bool silently coerced through `cast`. Regression tests `testCodegenSetParameterIntFamilyCoercesFloatAndString`, `testCodegenSetParameterIntFamilyRejectsUnparseableString`, `testCodegenSetParameterIntFamilyRejectsBool`, `testCodegenSetParameterStringRejectsNonString`, `testCodegenSetParameterFloatCoercion` in `UIRichInteractiveHelperTest`.
- **Codegen `_fireRebuildListeners` fired on no-op setter calls** — typed setters unconditionally called `_fireRebuildListeners()` at the end, even when the incoming value matched the stored value. With N cards in a hand, repeated `setParameter("status", "normal")` on hover-leave (a hot path in `UIRichInteractiveHelper`) triggered quadratic `UIScreen.addInteractives` rescans. Setters now short-circuit at the top: `if (this._<param> == v) return;` before the field write and downstream `_applyVisibility` / `_updateExpressions` / `_fireRebuildListeners`. Bool guard compares after Int conversion (`true/false` → `1/0`), so repeated bool sets also skip. Reference-equality for Array/Tile params is acceptable (callers only pass fresh instances when they mean to rebuild). Regression tests `testCodegenNoOpSetSkipsRebuildListener`, `testCodegenNoOpGuardCoversAllTypes`, `testCodegenNoOpEnumStatusGuard` in `UIRichInteractiveHelperTest`.
- **Codegen `setParameter` PPTBool string parsing only accepted `"true"`** — the generated dispatcher's Bool branch coerced String input via `(cast _value : String) == "true"`, silently returning `false` for `"yes"`, `"1"`, `"TRUE"`, or any typo. Inconsistent with `findEnumIndex` which accepts `true|yes|1|false|no|0` case-insensitive. Now aligned: `"true"|"yes"|"1"` → true, `"false"|"no"|"0"` → false (all case-insensitive), unknown strings throw explicitly (previously silently became false). Regression tests `testCodegenSetParameterBoolStringVariants`, `testCodegenSetParameterBoolUnknownStringThrows`, `testCodegenSetParameterBoolIntAndBoolPassthrough` in `UIRichInteractiveHelperTest`.
- **Codegen root-tint emitter still inlined alpha bake** — `ProgrammableCodeGen.generateFields` wrapped `$tintExpr` with `if (c >>> 24 == 0) c |= 0xFF000000` before `d.color.setColor(c)`, contradicting the strict-D rule that Int is final (runtime `applyExtendedFormProperties` writes the resolved int verbatim). Dormant because the `Std.isOfType(_, h2d.Drawable)` guard skips `h2d.Layers` / `h2d.Object` roots, but latent if any root ever became a Drawable subclass. Bake removed; codegen now calls `d.color.setColor($tintExpr)` directly.
- **LSP `tryStringToColor` 0x-branch baked alpha in stale compiled artifact** — `vscode/server/server.js` shipped with the pre-strict-D `var hexLen = s.length - 2; if (hexLen == 8) ...; if (color >>> 24 == 0) color |= -16777216;` logic, while the current Haxe source is a plain `return Std.parseInt(s)`. Users got "opaque red" diagnostics for `tint: 0xFF0000` from the LSP while the runtime builder stored it as transparent red. Rebuilt `lsp/lsp-server.hxml` and re-copied to `vscode/server/server.js` (the bump-but-don't-rebuild pattern in `install-local.ps1` caused this).
- **`Particles.emitBurstAt` over-counted rejected particles in `liveCount`** — the unconditional `liveCount++` fired before the `if (accepted)` gate, so a rejected-then-recycled particle (or any path through `init()` that returned `false`) permanently left `liveCount` +1 in looping groups. Inconsistent with `start()` / per-frame `emit()` which never increment `liveCount` on rejection. Moved `liveCount++` inside `if (accepted)`.
- **Codegen ignored root-level `pos:` / `scale:` / `rotate:` / `alpha:` / `tint:` / `filter:` / `blendmode:` on `@:manim` programmables** — `MultiAnimBuilder.startBuild` honours a programmable's root `pos:` (wrapping the result in a holder so user `setPosition` stays compositional with the offset) and applies all other extended form properties via `applyExtendedFormProperties(root, rootNode)`, but `ProgrammableCodeGen.generateFields` only walked `rootNode.children` and never read any of these. Same `.manim` rendered at full position/scale/alpha/etc. via the runtime path and at the identity transform via codegen. Fix is two-part: (1) when `rootNode.pos != ZERO`, codegen now adds an internal `_root:h2d.Layers` field (h2d.Layers so layered children's `cast(parent, h2d.Layers)` succeeds), routes top-level children through it, and emits the position via the existing `generatePositionExpr` / `ensureHexLayoutIfNeeded` pipeline (so grid/hex-coord-based root positions work too) — children added to `_root` instead of `this`, matching the runtime's holder→retRoot wrapping so user `inst.setPosition(X, Y)` now composes additively with the `.manim` root pos in both paths. (2) Root-level `scale` / `rotate` / `alpha` / `blendMode` / `tint` / `filter` are emitted into the constructor targeting either `this._root` (when wrapped) or `this` directly (when not), matching the runtime's behaviour where `applyExtendedFormProperties` runs on the inner root regardless of whether a holder was needed. `$param`-dependent values are tracked through `expressionUpdates` so `setParameter()` re-fires the property. New test 105 (`rootProperties`) exercises all five non-tint/blend properties at once. Test 78 (`characterSheetDemo`) had a workaround setting `instance.x = 50; instance.y = 80` to compensate for the missing offset; workaround removed and the test now passes at 100% builder/macro parity.
- **`UICardHandHelper` cards with internal `@switch` arms had stale bindings** — the card hand's private `interactiveHelper` (a separate `UIRichInteractiveHelper` instance, distinct from the screen's `autoStatusHelper`) was registered once per card at `buildCardEntry` time and never resynced. Cards whose programmable contained `@switch` arms with different interactive ids per arm would have stale bindings after a parameter change — drag/hover/click on the new arm's interactive was not routed to the helper's state machine. Fix: `buildCardEntry` installs a per-card rebuild listener on the card's `BuilderResult` that calls `interactiveHelper.resync(result, interactiveId)` on every rebuild. A new `unregisterCardEntry(entry)` helper consolidates the `interactiveHelper.unbind` / `screen.removeInteractives` / `removeRebuildListener` cleanup, called from all three card removal sites (`discardCard`, `clearHand`, card-played handler on `onMouseRelease`). Two regression tests in `CardHandIntegrationTest`: `testCardHandRebuildListenerResyncOnSwitchArmFlip` (flips card's internal switch arm and verifies both screen wrappers and `interactiveHelper` bindings resync), `testCardHandRebuildListenerRemovedOnDiscard` (verifies the listener is dropped on discard so subsequent `setParameter` on the stale result is a safe no-op).
- **`@switch` arm rebuild leaked stale per-element bookkeeping** — when a `@switch` arm flipped, the rebuild closure in `MultiAnimBuilder` did not clean up the `IncrementalUpdateContext`'s per-element collections (`trackedExpressions`, `conditionalEntries`, `conditionalApplyEntries`, `dynamicRefBindings`, `dynamicNameBindings`, `deferredEntries`, `activeTransitionTweens`) for entries whose underlying `h2d.Object` lived inside the destroyed arm container. Entries became inert (skipped via `isEffectivelyVisible`) but stayed in their arrays forever, growing unboundedly across repeated arm flips and pinning detached scene-graph subtrees in memory. Same bug class for the param-dependent `repeatable` rebuild closure. Fix introduces `IncrementalUpdateContext.cleanupDestroyedSubtree(ir, container)` which walks every per-element collection and removes entries whose `h2d.Object` is `container` itself or a descendant. SWITCH/REPEAT closures call it before `removeChildren()` so parent links are still intact for the descendant walk. Also drops `dynamicRefBindings` whose `childContext` belongs to a removed dynamicRef result (parallel to `rebuildDynamicNameRef`'s existing cleanup), and cancels `activeTransitionTweens` on doomed objects so transition completion callbacks don't fire on dead h2d.Objects. Dev-only inspection accessors (`getTrackedExpressionsCount`/`getConditionalEntriesCount`/`getDynamicRefBindingsCount`/`getDynamicNameBindingsCount`) added under `MULTIANIM_DEV`. Three regression tests in `BuilderUnitTest` (dev-only): `testIncrementalSwitchTrackedExpressionsCleanedUp`, `testIncrementalSwitchConditionalEntriesCleanedUp`, `testIncrementalSwitchDynamicRefBindingsCleanedUp` — each flips arms 10 times and asserts the relevant counter returns to its initial value.
- **Screen-side `UIInteractiveWrapper` and `autoStatus` bindings went stale after `@switch` arm flip** — `UIScreen.addInteractives(result, prefix)` snapshotted `result.interactives` once and never re-synced, so any switch arm flip that introduced new interactives (or removed old ones) left the screen's `interactiveMap`/`interactiveWrappers`/`autoStatusHelper.bindings` pointing at the previous arm's MAObjects. `screen.getInteractive("idInNewArm")` returned `null`, hover/press events were lost, and panel/tooltip helpers couldn't find the new interactives. Fix installs a rebuild listener on the BuilderResult's incremental context inside `addInteractives` (tracked in a new `interactiveSubscriptions` array). On each rebuild, the listener calls `syncInteractivesFrom(r, prefix)` which diffs the wrappers (removes ids not in `result.interactives`, adds new ids via `addInteractive`) and calls `autoStatusHelper.resyncAutoStatus(r, prefix)` to update auto-bindings. `removeInteractives(prefix)` and `clear()` drop matching subscriptions to prevent leaks. Existing autoStatus binding state (Normal/Hover/Pressed/Disabled) is preserved across resync — `UIRichInteractiveHelper.resync` now only `bind()`s ids that don't already exist (the previous attempt that called `registerInternal` reset live bindings to Normal and broke 10 existing autoStatus tests). Four screen-level tests in `UIRichInteractiveHelperTest`: `testScreenAutoResyncOnSwitchArmFlip`, `testScreenAutoResyncBackAndForthNoAccumulation`, `testScreenAutoResyncRemovedOnRemoveInteractives`, `testResyncPreservesExistingBindingState`. Manually-registered `UIRichInteractiveHelper` instances (e.g. `UICardHandHelper.interactiveHelper`) are NOT auto-resynced — users must call `helper.resync(result, prefix)` explicitly.
- **`@switch` arm rebuild dropped registrations into a throwaway `InternalBuilderResults`** — the SWITCH and param-dependent REPEAT rebuild closures in `MultiAnimBuilder.build()` constructed a fresh local `ir = {names: [], interactives: [], slots: [], dynamicRefs: new Map(), htmlTextsWithLinks: []}` and passed it to the recursive `build()` call instead of the parent's `internalResults`. Anything inside the new arm or new repeat iteration (named elements, `interactive(...)`, `#name slot`, `dynamicRef`, `htmlTextsWithLinks`) was registered into the throwaway and orphaned: `result.getInteractive(id)` / `result.getSlot(name)` / `result.getDynamicRef(name)` / `result.getUpdatable(name)` / `result.hasName(name)` all returned `null`/threw for content inside the new arm or iteration. Tooltip/panel/rich-interactive helpers stopped seeing those interactives, autoStatus auto-wiring lost track of the new arm. Fix introduces `MultiAnimBuilder.removeRegistrationsUnder(ir, container)` (descendant scan that drops stale entries before rebuild) and routes both rebuild closures through `IncrementalUpdateContext.cleanupDestroyedSubtree(ir, container)` (which calls `removeRegistrationsUnder` plus the per-element bookkeeping cleanup described above), then passes the parent `internalResults` to the rebuild. Five builder-level regression tests in `BuilderUnitTest` covering each affected collection type: `testIncrementalSwitchInteractiveVisibleAfterArmSwitch`, `testIncrementalSwitchSlotVisibleAfterArmSwitch`, `testIncrementalSwitchDynamicRefVisibleAfterArmSwitch`, `testIncrementalSwitchNamedElementVisibleAfterArmSwitch`, `testIncrementalRepeatableInteractiveCountChange`, plus two stale-entry guards (`testIncrementalSwitchNoStaleAccumulation`, `testIncrementalRepeatableInteractiveShrinkNoStale`). Codegen `rebuildSwitchArmByOrdinal` was verified to have no id-based interactive consumers (codegen instances expose typed `_eN:MAObject` fields only, no `getInteractive(id)` accessor) and is left as-is.
- **`@switch` pipe arms on color/int/bool params** — pipe-separated arms (`#FF0000 | #00FF00: ...`, `0xFF | 0x10: ...`, `true | false: ...`) stored the raw lexeme strings in `CoEnums`, while the runtime parameter is an integer — `matchSingleCondition` compared `a.contains(Std.string(currentValue))` (e.g. `"16711680"` vs `"#FF0000"`) and no pipe arm ever matched. Parser now routes each pipe value through `stringToConditional(v, paramType)` and wraps the results in a new `CoAnyOf` variant; builder and codegen both handle `CoAnyOf` (including negation via `CoNot` and param-ref collection). Enum/string params keep `CoEnums`. Two incremental-mode regression tests in `BuilderUnitTest`: `testIncrementalSwitchPipeColor`, `testIncrementalSwitchPipeHexInt`.
- **`setParameter` on `:color` param stored un-baked alpha** — `IncrementalUpdateContext.setParameter` stored `int` values directly without calling `addAlphaIfNotPresent`, while `.manim` literals (`#FF0000`) are baked to `0xFFFF0000` at parse time. `@switch`/`@()` arms then compared the baked literal against the un-baked runtime value and no arm matched. `setParameter` now bakes alpha for `PPTColor` params so set-equivalence matches parser default-equivalence.
- **`0xRRGGBB` color literals in `.manim` source stored un-baked alpha** — the parser's `tryStringToColor` `0x`-prefix branch returned `Std.parseInt(s)` as-is, so `tint:color=0xFF0000` stored `0x00FF0000` (alpha 0) while `tint:color=#FF0000` stored `0xFFFF0000`. Two `.manim` sources that looked equivalent produced different runtime values, and `@switch`/`@()` arms never matched `0x`-default programmables. Fix is length-aware on the source string (before `Std.parseInt` collapses leading zeros): 6-or-fewer-digit literals (`0xFF0000`) route through `addAlphaIfNotPresent` and bake `0xFF` alpha to match `#RRGGBB`; 8-digit literals (`0x80FF0000`, `0x00FF0000`) are preserved verbatim, so users can still express explicit alpha including fully-transparent colors. Regression tests `testParser0xColorLiteralAlphaBaking`, `testParser0xColorLiteralExplicitAlphaPreserved`, and `testParser0xColorLiteralExplicit8DigitZeroAlpha` in `BuilderUnitTest`.
- **Codegen setter/constructor on `:color` param stored un-baked alpha** — sibling of the above for the `@:manim` codegen path. The generated `setTint(v:Int)` assigned `v` raw, and the instance constructor forwarded `tint` into `_tint` without baking. `@switch(tint) { #FF0000: ... }` therefore never matched when the caller passed `instance.setTint(0xFF0000)` or `factory.create(0xFF0000)` — the arm condition compared against the parser-baked `0xFFFF0000`. Fixed at the type level: `PPTColor` storage fields now have type `bh.base.ColorUtils.ColorArg` (a new `abstract ColorArg(Int) from Int to Int` wrapper whose `@:from Int` calls `addAlphaIfNotPresent`). Every `_<name> = <int>` assignment — constructor forward, setter write, any future write site — goes through `fromInt` automatically; reads are zero-cost via `to Int`. Public arg types stay `Int` so default argument values remain literal constants (Haxe forbids cast / abstract expressions as arg defaults). Regression tests `testCodegenSetColorParamAlphaBaked` / `testCodegenCreateColorParamAlphaBaked` in `ProgrammableCodeGenTest`.
- **DevBridge HTTP DoS safeguards** — the JSON-RPC server's `HttpConnection` read headers and body with no upper bound and had no idle timeout. A malicious or stuck client could park a socket indefinitely and grow `headerBuf`/`bodyBuf` without limit. Now enforces `MAX_HEADER_BYTES=64KB`, `MAX_BODY_BYTES=16MB`, and `IDLE_TIMEOUT_SEC=30`, with `431 Request Header Fields Too Large`, `413 Payload Too Large`, and `408 Request Timeout` responses. `ScreenManager.update()` calls `devBridge.tick()` each frame to close expired pending connections. Header scanning also only re-scans the unread tail (`headerSearchFrom`) so parse cost is O(header size) not O(n²). All header/body reads now use `readBytes` into a single bounded buffer instead of byte-at-a-time.
- **`crossfade` transition with non-linear easing** — the sequential hide→show pivot was encoded as a `Custom` tween that tested `v <= 0.5` inside its setter, but `Tween.step` feeds `easedT = easingFn(t)` into the setter, not raw time. With `easeOutQuad + duration=0.3`, `easedT=0.5` at raw `t≈0.293` so the fade-in started at ~30% of `2·duration` instead of 50%, making the show phase ~70% of total time while the hide phase stayed at 50%. With `easeInQuad` the asymmetry reversed. Fix: show branch now uses `tm.sequence([pause, fadeIn])` so the pause is raw-time and the easing only applies to the fade-in half. Applied to both `CodegenTransitionHelper` paths and `MultiAnimBuilder`'s incremental transition. A new `transCrossfadeEased` row was added to `test/examples/95-transition` to guard the fix (the previous linear-only coverage couldn't detect the skew).
- **`flipX` / `flipY` transitions were concurrent, not sequential** — the show branch used `tm.tween(halfDuration, …)` starting immediately, so the hide on the outgoing element and the grow on the incoming element ran in parallel over the same `halfDuration` instead of hide-then-show. At any mid-point both elements were visible at partial scale. Fix: show branch now uses `tm.sequence([pause, grow])` with `halfDuration` for each phase, yielding the intended "old shrinks, then new grows" behavior over total `duration`.
- **`@switch` codegen `paramFieldExpr` prefix** — codegen's `paramFieldExpr()` used `"_p_"` prefix for param field references but param fields are created with `"_"` prefix, causing compile-time errors when a codegen programmable used `@switch` on its own parameters. Fixed to use `"_"`.
- **`@switch` nested param ref collection** — `collectNodeParamRefs` did not handle `SWITCH` or `REPEAT` node types, so nested `@switch` params and repeatable count params inside switch arms were not tracked. Changing the inner switch param or repeatable count would not trigger the outer switch rebuild. Both are now collected.
- **Incremental `applyConditionalChains` O(N²) → O(N)** — `resolveVisibilityForChildren` used linear scans of `conditionalEntries` and `conditionalApplyEntries` for every child node. Now builds `StringMap` lookup tables keyed by `uniqueNodeName` before the recursive walk.
- **`@switch` on `color`/`string`/`bool`/`int` params** — see "Changed" above. The single-value-arm fix closes a latent bug where `@switch` on a `bool` parameter silently never matched any arm (`CoEnums(["true"])` vs runtime `Value(1)` → string compare `"1" != "true"`).
- **`staticRef`/`dynamicRef` on `:bool` parameters** — `staticRef($prog, enabled=>true)` used to throw `expected integer, got "true"` at build time because `updateIndexedParamsFromDynamicMap` routed `PPTBool` through `resolveAsInteger`, which cannot parse the literal `true`/`false` string. Now routes through `resolveAsBool` (which handles `RVString` via `tryStringToBool`). Affected any code passing bool literals to `:bool` programmable params from `.manim`; the pre-existing `:[true,false]` enum workaround is no longer required.
- **HTML visual test report sort order** — tests numbered 100+ were interleaved with tests 10/11/... in the report because `reserveOrderIndex()` reflected utest's string-sorted method discovery order (`"test100_*" < "test11_*"`). Sort now prefers the numeric test number extracted from the `#NNN: name` display format, falling back to `orderIndex` only as a stable-compare tiebreaker for un-numbered entries.
- **Particle `advanceTime` accumulator** — when `externallyDriven: true`, multiple `advanceTime(dt)` calls between renders now accumulate per-call `dt` into `_externalDt` instead of overwriting it. Emission still runs per call (each with its own `dt`), and the accumulated total is consumed as a single physics step at the next `draw()`. Previously, all but the last sub-step's physics were dropped when the game advanced time multiple times per frame.
- **`.anim` `flipX`/`flipY` frame-size consistency** — validates at parse time that all frames share the same untrimmed size when `flipX`/`flipY` is enabled (extrapoints mirror around a single origW/origH, so mixed-size frames would desync points from visuals). Throws a clear error listing the mismatched dimensions. Extrapoint mirror math also uses `Math.round(origW - 2 * cx - p.x)` instead of `Std.int(origW - 2 * cx) - p.x` — the old form truncated the half-extent before subtracting `p.x`, losing fractional precision when `origW - 2 * cx` wasn't integral.
- **Tile source `center()`/`pivot()` nesting validation** — rejects nested pivot/center wrappers (`center(pivot(...))`, `pivot(0, 0, center(...))`) at parse time because the outer wrapper would silently override the inner one. Also validates `pivot(x, y, ...)` ratios are in 0..1.
- **`ScreenShakeHelper` baseline capture** — the previous implementation captured `target.x/y` at construction time and snapped back to it on every frame, which fought any gameplay motion of the target (camera scroll, character follow, etc.) and also snapped to stale coordinates if the target moved after construction. Now uses per-frame delta tracking (`target.x += totalX - lastDx`), so gameplay motion is preserved; `stop()` removes only the shake's residual offset. The unused `setOrigin(x, y)` method is removed. Jitter switched from a time-seeded integer hash to `hxd.Rand` for proper uniform angle distribution.
- **`.anim` tile center mutation** — `AFtoFrame` now clones the tile before applying center offsets (`dx`/`dy`), preventing mutation of shared cached atlas tiles. Previously, setting `center:` in an `.anim` file would mutate the atlas tile in-place, causing offset corruption when the same tile was reused across multiple animations or reloads.
- **`UICardHandHelper.dispose()` listener leak** — `dispose()` now clears `chainedListeners`, preventing stale references from accumulating after card hand teardown.
- **Negative hex integer literal in `EAny` context** — `parseAnything()` merged `TInteger` and `THexInteger` cases and called `stringToInt(n)` without prepending `"0x"`, so literals like `-0xFF` used anywhere an "any value" was expected (e.g. interactive metadata: `interactive(..., threshold => -0xFF)`) resolved to `0` instead of `-255`. Hex and decimal are now split into separate branches. `parseIntegerOrReference` already handled this correctly.
- **Unterminated double-quoted string silently accepted** — `MacroLexer`'s double-quoted string path lacked an EOF check: when the closing `"` was missing, the loop fell off the end of input and returned a `TQuotedString` holding partial content, producing confusing downstream errors several tokens later. Now throws `Unterminated string, missing closing double quote` with the file/line/column of the opening quote, matching the single-quoted path.
- **`.anim` filter `@else` / `@default` stacking with matching `@(cond)`** — `resolveAnimFilters` accepted every entry whose state selector didn't actively contradict the runtime selector, so `@else` / `@default` fallbacks (which parse to effectively-empty selectors that always match) stacked on top of the matching `@(cond)` entry instead of being mutually exclusive. Concretely: `@(level >= 3) outline` followed by `@else pixelOutline` would apply *both* filters at `level=5`, and `@(team=>red) tint: #FF0000` followed by `@default tint: #FFFFFF` would always end up white because the default ran last and overwrote the match. Parser now tags alternation chains with a `group` id (a new `@(cond)` starts a group; subsequent `@else`/`@else(cond)`/`@default` siblings join it; an unconditional entry breaks the chain and is always applied). `resolveAnimFilters` picks the best-scoring entry per group while still stacking unconditional filters on top. Five regression tests in `AnimFilterStateConditionalTest`.
- **`@switch` arm + parameterized slot build context inheritance** — `rebuildSwitchArmByOrdinal` and `buildSlotContent` both built their children with `cast null` for grid/hex coordinate systems and a fresh `{callback: defaultCallback}` instead of inheriting the caller's context. Two latent bugs in both paths: (B1) an arm or parameterized-slot body that used `$grid.pos(...)` / `$hex.*` inside a nested element (e.g. `graphics(rect(...): $grid.pos(0, 0))`) crashed with `gridCoordinateSystem is null` on initial build and every rebuild, because the programmable's `grid:`/`hex:` declaration was not propagated into the nested build. (B2) a `placeholder(..., callback("name"))` inside an arm silently lost the user's `BuilderParameters.callback` after any `setParameter()`-driven rebuild, because the rebuild installed `defaultCallback` instead. Fix captures `builderParams` and resolves the enclosing node's grid/hex systems via the parent chain *before* `pushBuilderState`, then threads them into the nested `build()` calls in both `rebuildSwitchArmByOrdinal` and `buildSlotContent`. `placeholderObjects` and `scene` fields are also inherited. Three codegen regression tests in `ProgrammableCodeGenTest`: `testCodegenSwitchArmGridContext`, `testCodegenSwitchArmCallbackInheritance`, `testCodegenSlotParamGridContext`.
- **LSP `.anim` context detection in comments/strings** — `AnimAnalyzer.detectContext` walked every character of every line counting `{`/`}` and keyword-matched line starts without comment or string awareness. Braces inside `//`/`/* */` comments shifted the nesting depth, and keywords inside strings or comments (e.g. `// playlist disabled`) flipped the completion context to the wrong block, corrupting hover/completion results for any `.anim` file with prose. Lexer now strips line comments, tracks block comments across lines, and skips `"..."` string literals (including `\"` escapes) before counting braces. Keyword detection is suppressed while inside a block comment.
- **Particle `emitFilter` sub-emitter cascade** — when `emitFilter` rejected a particle, the particle's `OnBirth` trigger still fired during the next `init()` step and, once its forced-dead lifecycle tripped, its `OnDeath` trigger fired as well, producing an explosive spawn loop at the rejection boundary (every rejected particle spawned a full child burst plus a death burst). `Particle.rejected` now flags filter-rejected particles; `init()` clears it on every fresh spawn and `emitFilter()` sets it when discarding. Both `OnBirth` (delayed spawn step) and `OnDeath` (lifecycle expiry) sub-emitter triggers skip rejected particles.
- **Particle recycle double-init** — when a particle's lifecycle expired on a looping group with live headroom, `init()` ran to recycle the particle but `timeNormalized` (computed earlier in the same update step from the pre-init `life`) was still > 1, so the lifecycle branch immediately re-fired `OnDeath` triggers and potentially re-ran the recycle path with stale state. The recycle branch now returns `true` immediately after `init()`, skipping the stale lifecycle check for that frame.
- **Particle shutdown terminal multipliers** — when `shutdown()`'s curve reached `rate >= 1.0` and `shutdownTargetCount <= 0`, the shutdown flag was cleared so remaining particles could die naturally, but `shutdownAlphaMult`/`shutdownSizeMult`/`shutdownSpeedMult` still held their terminal (~0) values from the final curve sample. Any live particles finishing their lifetime after shutdown exit rendered at 0 alpha/size/speed instead of their natural values. All three multipliers are now reset to `1.0` at the moment `shutdownActive` flips to false.
- **Particle `removeGroup()` batch leak** — `Particles.removeGroup(id)` only removed the group entry from the map and left the underlying `h2d.SpriteBatch` attached to the scene with its particles. The batch and its elements stayed resident and continued rendering until the parent `Particles` object was disposed. Now clears and removes the batch before dropping the map entry.
- **Particle `tiles.length == 0` early return** — `init()` previously `return`ed from the middle of the function when a group had no tiles (waiting for async asset load), skipping the remainder of the setup (`p.vx *= speed`, gravity/acceleration/rotation assignment, `emitFilter()` check). First-frame particles from tile-less groups started with unscaled velocity and bypassed `emitFilter`. Now wraps tile selection in a conditional so the rest of init always runs.
- **`ProgrammableBuilder.setFontFromMultiple` empty-fonts crash** — when the font fallback array was empty (no best match and no fonts at all — e.g. misconfigured rich text), `fonts[fonts.length - 1]` evaluated to `fonts[-1]` which is `null` on HashLink and could crash further downstream. Now retains the original font when the array is empty instead of clobbering it with null.
- **`.anim` conditional duplicate state key** — `@(level >= 3, level <= 5, level != 0)` silently overwrote prior entries for the same state name in the conditional map, so only the last comparison for a given state was applied. Parser now rejects duplicate state keys in a single conditional with a clear error.
- **`.anim` negative integer comparison literals** — `@(level >= -1)`, `@(xp > -100)`, etc. failed to parse because `expectIdentifier()` didn't accept a leading minus. Comparison conditionals now use `expectSignedIdentifier()`, matching the negative-literal handling already in place for plain value lists.
- **`.anim` parser `throw` → `syntaxError`** — several late-phase validation errors (`no playlist for state X`, `animation X not reachable`, `Extra point X in anim Y not reachable`) were raised via raw `throw` statements, bypassing the parser's structured error reporting (file/line/column context). All now route through `syntaxError()` for consistent strict-mode output.
- **`@switch` on `:array` params** — `@switch` on an array-typed parameter previously hit the `default:` branch of the type check and silently emitted string arms that could never match. `PPTArray` now surfaces a clear "Array cannot be matched — @switch requires a discrete type" error at parse time.
- **`MacroManimParser` iterator error message** — the error listing valid iterator types for `repeatable()` / `tilegroup` named the leading option `grid`, but the parser rejects `grid` and accepts `step`. Corrected to list `step, layout, array, range, stateanim, tiles`.
- **`ManimKeywordInfo` missing `SWITCH` entry** — the element-ctor and node-type lists used by the LSP's exhaustive-switch compile-time guard did not include `SWITCH`, so any addition to those enums that touched `SWITCH` would have failed the exhaustive check silently. Added `SWITCH` to both lists.
- **`UICardHandHelper` arrow ignoring `canPlayCard`** — while dragging a card into a targeting zone, the arrow's valid/invalid color (green vs red) was based solely on whether the cursor hovered a registered target interactive. A `canPlayCard` veto (e.g. "this shuttle can't board that tile") silently refused the drop but the arrow still lit green during the drag, giving the player no feedback. `updateCardDrag()` now calls `canPlayCard(cardId, currentResult)` each frame and forces `targeting.forceValid` to match, so invalid drops flash red before release.

### Fixed
- **Parser: `@switch` range with non-integer values silently mapped to 0** — `parseRV()` now throws a parse error for non-numeric values (e.g., `foo..10`) instead of silently treating them as 0.
- **Parser: `@switch` parameter name not validated** — `@switch(paramName)` now validates that `paramName` exists in the programmable's parameter definitions, matching the behavior of `@()` conditionals.
- **Parser: data block forward-referenced types silently misclassified** — enum/record types referenced before their definition in a `data {}` block now produce a clear error instead of being silently treated as a different type.
- **Draggable: disable during drag orphans state** — setting `enabled = false` while dragging now calls `cancelDrag()` via property setter, properly restoring capture, alpha, layer, origin, and source slot instead of leaving the drag orphaned.
- **Draggable: zero-distance animation skip ignores visual effects** — `startAnimation()` no longer skips zero-distance paths when `animApplyScale`, `animApplyAlpha`, or `animApplyRotation` flags are set, allowing in-place fade/scale/rotation animations.
- **Draggable: `swapMode = true` without `sourceSlot` silently degrades** — now throws with a clear error message via property setter validation.
- **CardHand: invalid path names deferred to animation time** — `drawPathName`, `discardPathName`, `returnPathName`, `rearrangePathName` are now validated at construction via `builder.hasNode()`, failing fast with a descriptive error instead of throwing during the first animation.
- **Grid cell drag: source cell shows stale visual during drag** — `cellDragStart` now rebuilds the source cell with null data to show the empty slot visual while dragging, instead of leaving the previous cell appearance.
- **Grid cell drag: swap target rebuilds too early** — target cell visual rebuild is now deferred until the snap animation completes, preventing a flash of the new content while the drag visual is still animating on top.
- **Grid cell drag: source data lost on cancel** — `cellDragRebuildSourceAndFinish` now restores the original cell data before rebuilding, fixing an issue where cancelling a drag left the source cell empty.
- **CardHand: layout path cache not invalidated on hot-reload** — `wireCardHandReload()` now calls `invalidateLayoutCache()` so PathLayout mode picks up changed `.manim` paths after reload.

- **Unterminated block comments detected** — both `.manim` and `.anim` parsers now throw a clear error (`Unterminated block comment, missing closing */`) instead of silently dropping content after an unclosed `/*`.
- **Division/modulo by zero** — `MultiAnimBuilder` now throws `Division by zero` / `Modulo by zero` for `/`, `div`, `%` operators in both integer and float expression paths, instead of crashing (HashLink) or producing `Infinity`/`NaN`.
- **Incremental mode evaluates expressions in non-visible conditional nodes** — deferred build pattern now skips expression evaluation entirely for conditional nodes that don't match, matching non-incremental/codegen behavior. Previously, incremental mode built all nodes and toggled visibility, which caused division-by-zero crashes when expressions referenced parameters guarded by range conditionals (e.g., `$baseShield * 36 / $maxShield` inside `@(maxShield=>1..99)`). Deferred nodes materialize on first visibility with current parameter values and register tracked expressions for future updates. Hidden tracked expressions are also skipped during `applyUpdates()`.

- **Card hand callback chaining for multi-grid targeting** — `registerAsCardTarget()` now chains accepts/highlight callbacks instead of overwriting, so multiple grids can coexist as card targets on the same card hand.
- **Rich text XML character escaping** — `TextMarkupConverter` now escapes `<`, `>`, `&` characters before markup conversion, preventing `h2d.HtmlText`'s XML parser from misinterpreting literal angle brackets and ampersands in text content (e.g., `"Hull<25%"`, `"fire & ice"`).
- **Custom filter `$param` type validation** — `$param` references in custom filter arguments now skip type validation at parse time (type is resolved at build time). Previously, passing a `$color` parameter to a `CFColor` argument would incorrectly fail validation.
- **Codegen `WITH_OFFSET` for non-static bases** — `generatePositionExpr` now correctly handles `.offset()` on non-static coordinate expressions (e.g. `$ref.extraPoint(...).offset(x, y)`). Previously returned null when the base couldn't be resolved at compile time, causing elements to be positioned at (0, 0).
- **Conditional Graphics content loss on re-show** — `h2d.Graphics` elements inside conditional blocks (`@()`, `@if`, `@else`, `@default`) lost their draw commands when toggled off and back on, because Heaps' `onRemove()` clears Graphics content on scene graph removal. Fixed by always registering a redraw tracked expression for Graphics in incremental mode, re-firing tracked expressions when `addToGraph` re-adds an element, and keeping deferred entries without param refs for full re-materialization on re-show.
- **`safeDetach` for h2d.Graphics reparenting** — extracted `HeapsUtils.safeDetach()` utility that detaches objects without triggering `onRemove()` cascade (which destroys `h2d.Graphics` draw commands). - **`detachCellVisual` immediate rebuild** — `detachCellVisual()` now rebuilds the cell entry immediately after detaching, fully severing the detached object from the grid. Prevents `rebuildCell()` from removing the detached object.
- **Grid `dispose()` fires swap onComplete callbacks** — `dispose()` now fires `onComplete` on active swap animations before removing them, preventing game logic from being left in an inconsistent state.
- **Path `toPoints()` off-by-one** — curve segment `toPoints()` now generates `steps + 1` points, including the endpoint. Previously the last point was missing, causing visual gaps at the end of arc, bezier, spiral, and wave segments.
- **Parser `namedCoordSystems` leak across programmables** — `namedCoordSystems` (named `grid:`/`hex:` systems) is now reset on each root-level node parse, preventing systems from one programmable leaking into siblings.
- **ProgrammableBuilder missing animation error** — `constructStateAnims()` now throws a descriptive error (`Animation "X" not found in sheet "Y"`) instead of null-reference crash when a state animation references a missing sheet animation.
- **CardHand cursor not restored on `clearHand()`** — `clearHand()` now restores the system cursor to `Default` when called while targeting mode is active with `hideCursorWhileTargeting`.
- **PanelHelper named panel fade-out race** — `closeNamed()` now tracks in-flight fade-out tweens and cancels the previous one before starting a new close, preventing orphaned scene graph objects when a named panel is rapidly closed/reopened.
- **ScrollableList `currentItemIndex` error message** — corrected bounds description from `[0..length]` to `[-1..length-1]` to match the actual valid range.
- **DevBridge SSE broadcast flag reset on error** — `broadcastSseEvent()` now wraps serialization in try/catch so `sseBroadcasting` is always reset, preventing a `Json.stringify` failure from permanently blocking SSE events.
- **CardHand `applyCardToCardEffects()` bounds guard** — `indexOf` result is now checked for -1 before array access, preventing out-of-bounds access if the target card was removed externally.
- **Repeatable `$i` conditions not re-evaluated by `setParameter`** — conditions inside repeatables comparing the loop variable against a parameter (e.g., `@($i < $level)`) now correctly update when that parameter changes via `setParameter()`. Two fixes: (1) `CoRange` comparison/range bounds are now stored as `ReferenceableValue` and resolved at build time, so `$param` references in `@($i < $param)`, `@(param => $from..$to)`, and `between $from..$to` work instead of being baked to 0 at parse time; (2) incremental tracked expressions for repeatables now include parameter references from children's conditions, triggering a rebuild when those parameters change.
- **Parser 2D index variable null guard** — `#name[$x, $y]` parsing now checks `scopeVars == null` before the second variable lookup, matching the first variable's defensive check.
- **Touch input broken in UIRichInteractiveHelper** — `UIPush` handler now transitions from both `Normal` and `Hover` states to `Pressed`, fixing touch input which sends `UIPush` without prior `UIEntering`.
